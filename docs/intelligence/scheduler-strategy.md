# Scheduler Strategy

How and when the daily hospitality intelligence cron fires.

**Last refreshed:** 2026-05-12
**Decision:** Vercel Cron (single daily firing). Supabase pg_cron evaluated and parked as future option.
**Phase 2.6 status:** Cron now hydrates real T2 cookie jars per run · validates session health against the canonical target · auto-degrades to anon mode on validation failure (T2 marked `refresh_failed`, Admin UI red banner triggers immediately). Smoke-tested 2026-05-12 · 9/10 sources success · 114 articles ingested · 34 Hosteltur authed body fetches.

---

## 1. Requirements recap

- **Once per day, never more**
- **Local time**: 08:48 Europe/Madrid
- **Owner**: HotelVALORA platform (not Supabase backend)
- **Observability**: First-class logging, retry visibility, easy on-call review
- **Cost**: Free tier (Vercel Hobby)
- **Failure isolation**: One source breaking doesn't abort the whole run

## 2. Vercel Cron vs Supabase pg_cron

| Dimension | Vercel Cron | Supabase pg_cron |
|---|---|---|
| Trigger surface | HTTPS request to a Next.js route | SQL function call (or `net.http_get`) |
| Timezone awareness | UTC only — DST handled by adjusting the expression | Postgres `timezone` setting — true TZ support |
| Observability | Vercel runtime logs + dashboard | Postgres logs + `cron.job_run_details` table |
| Local debugging | `curl localhost:3000/api/cron/...` with `CRON_SECRET` header | Requires running the SQL function manually |
| Code path | TypeScript in the Next.js app (same toolchain as the rest) | SQL/PLpgsql or HTTP-call workaround |
| Fail isolation | `Promise.allSettled` over sources | Manual error handling in PL/pgSQL |
| Cost (Hobby tier) | 2 cron jobs free | Free with Supabase |
| Co-location with consumer code | ✅ Same repo, same deploy pipeline | ❌ Lives in DB layer, separate dev cycle |

**Decision: Vercel Cron** — better DX, observability, and code co-location. pg_cron remains a fallback if we ever need pure-database scheduling (e.g., a `pg_notify` daily fan-out to Supabase Realtime subscribers).

## 3. Cron expression for 08:48 Europe/Madrid

Vercel Cron only accepts UTC expressions. Spain observes daylight saving time:

| Period | Spain TZ | UTC offset | Cron expression for 08:48 local |
|---|---|---|---|
| Winter (late Oct → late Mar) | CET | UTC+1 | `48 7 * * *` |
| Summer (late Mar → late Oct) | CEST | UTC+2 | `48 6 * * *` |

**Adopted compromise**: schedule at `48 7 * * *` UTC year-round. Result:
- Winter: fires at **08:48 Europe/Madrid** ✅
- Summer: fires at **09:48 Europe/Madrid** (1-hour drift, intentionally accepted)

Rationale: hospitality news isn't intraday-sensitive. A 1-hour seasonal drift in ingestion time is invisible to the product. Avoiding manual DST swaps keeps ops simple.

### Future option: dual cron entries
Vercel Hobby allows 2 cron jobs. We could schedule both `48 7 * * *` and `48 6 * * *` UTC and gate firing inside the handler by reading the current Madrid time. Probably overkill — the 1-hour drift is acceptable and adding a second cron complicates the audit picture (which row in `news_ingestion_runs` is "the" daily run?).

## 4. Cron config file shape (Phase 2)

Lives in `apps/web/vercel.json` (NOT the repo-root `vercel.json` — that file is just a hint; the Vercel project's Root Directory = `apps/web`).

```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "crons": [
    {
      "path": "/api/cron/hospitality-intel",
      "schedule": "48 7 * * *"
    }
  ]
}
```

Vercel auto-creates the cron when this file is committed. To disable, remove the entry and redeploy.

## 5. Authentication

Vercel Cron sends a request with the header:

```
Authorization: Bearer <VERCEL_CRON_SECRET>
```

The secret is auto-generated by Vercel and exposed to the function as `process.env.CRON_SECRET`. The cron route handler MUST verify it:

```ts
if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
  return new NextResponse("Forbidden", { status: 403 });
}
```

Without this check, anyone could trigger the ingestion by hitting the URL. With it, only Vercel's cron infrastructure (and any operator who knows the secret) can fire it.

The `CRON_SECRET` env var doesn't need to be added manually to Vercel — it's auto-populated by the platform.

## 6. Manual triggering (ops)

During Phase 2 development and for ad-hoc re-runs:

```bash
curl -X GET https://www.hotelvalora.com/api/cron/hospitality-intel \
  -H "Authorization: Bearer $(vercel env get CRON_SECRET production)"
```

(That command requires the operator to have Vercel CLI access. Useful for re-running after a source recovers from downtime.)

## 7. Observability

### 7.1 · Per-firing
- Vercel runtime logs show one request to `/api/cron/hospitality-intel`
- Function logs print structured JSON per source (start, finish, counters)

### 7.2 · Per-source per-day
- `select * from news_ingestion_runs where run_started_at > now() - interval '24 hours' order by run_started_at desc` — single query for the full daily breakdown
- `status = 'failed'` rows are the on-call signal

### 7.3 · Health monitoring
Future Phase 4 wires up:
- Slack webhook on any source with 3 consecutive `failed` statuses
- Email alert if total daily corpus growth < 30% of trailing 7-day average

## 8. Cost

Vercel Hobby plan:
- Cron firings: **unlimited daily firings, capped at 100 jobs configured** — we use 1
- Function execution: 100 hours/month — we use ~30 minutes/month (1 firing × ~60s × 30 days)
- Bandwidth: minimal (fetches small JSON/XML/HTML, writes to Supabase, no client traffic)

**Effective cost: €0/month.** When the team grows past Hobby plan limits for unrelated reasons (analytics, more traffic), the cron remains free.

## 9. Disaster recovery

If Vercel Cron stops firing (platform incident or accidentally disabled):

1. `news_ingestion_runs` shows no rows for the missed day → operator notified by Phase 4 monitoring
2. Manual re-trigger via the curl command above
3. Re-runs are idempotent — `url_hash` constraint prevents duplicate inserts; `last_seen_at` and `occurrences` update correctly
4. If multi-day backlog: the regex categoriser is deterministic, so re-running over the corpus produces the same categories. Embeddings + AI summaries (Phase 4) only run for rows where `enriched_meta IS NULL`.

## 10. When to revisit this decision

| Trigger | Reconsider |
|---|---|
| Need >2 cron jobs (e.g., per-region batched fans) | Move to Vercel Pro OR migrate to pg_cron |
| Cron exceeds 60s execution | Move to Vercel Edge runtime OR Pro plan (higher cap) |
| Want true Europe/Madrid TZ (no DST drift) | pg_cron with `timezone='Europe/Madrid'` set on the cron's role |
| Phase 6 alerts engine fires reactively | Add Supabase Realtime broadcast — cron stays for the daily batch |
