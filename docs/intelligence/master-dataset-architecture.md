# Master Dataset Architecture

Why two canonical .xlsx workbooks today, and how they evolve into Supabase tables tomorrow.

**Last refreshed:** 2026-05-11

---

## 1. The decision

HOTELVALORA needs an institutional, version-controlled, human-editable corpus for transactions and projects **before** the runtime read path is wired into Supabase. We pick:

- **Two separate .xlsx workbooks** as the canonical source — one transactions, one projects
- **Generated by a checked-in Python script** (`services/transactions/scripts/build_masters.py`) so the schema is reproducible
- **Append-only** — never overwrite; supersedes via a new row with `supersedes_id`
- **Five sheets per workbook** — DATA · DICTIONARY · INGESTION_LOG · SOURCES_REGISTRY · README
- **Backed by Supabase rows in Phase 5** once enrichment + entity resolution stabilise

This is the institutional pattern: pen the contract in something the operator can audit (Excel), enforce shape via a generator (Python), and migrate the read path to a database (Postgres) once the operations layer earns it.

## 2. Why .xlsx, not Supabase, today

| Reason | Detail |
|---|---|
| **Operator legibility** | Hotel investment operators read Excel natively. Forcing SQL on the data-quality loop slows institutional adoption. |
| **Cell-level audit** | A row with a wrong price → operator opens Excel, edits, saves. Same workflow as their existing CoStar / brokerage workflows. |
| **No premature schema lock-in** | The first 90 days will reveal columns we haven't anticipated. Excel tolerates this; a Postgres schema makes it 3× more expensive to evolve. |
| **Reproducible** | The generator is the contract. Anyone can rebuild the workbook from `build_masters.py`. |
| **Git-friendly for the canonical** | Two .xlsx blobs in git are <500 KB combined. Operators can `git pull` to get the latest canonical. |
| **Zero infrastructure** | No migrations, no RLS, no service-role keys, no API surface. Phase 1 ships with file I/O only. |

## 3. Why two workbooks, not one

Mixing transactions + projects into a single workbook would force a `record_kind` column, ambiguous `*_uid` semantics, and a unified category enum that doesn't fit either domain. The two domains diverge on:

| Dimension | Transactions | Projects |
|---|---|---|
| **Primary axis** | Deal closing | Project stage progression |
| **Time anchor** | `closed_at` / `announced_at` | `announced_at` / `groundbreaking` / `estimated_opening` |
| **Pricing** | `price_eur`, `cap_rate`, `gop_per_key_eur` | `capex_eur`, `capex_per_key_eur` |
| **Parties** | `buyer_*` + `seller_*` | `developer_*` + `operator_*` |
| **Stage enum** | n/a | `announced`/`permitting`/`under_construction`/`pre_opening`/`opened`/`cancelled`/`on_hold` |
| **Categorisation** | `acquisition`/`sale`/`joint_venture`/`refinancing`/`distress` | `development`/`branded_residences`/`flex_living`/`pipeline_announcement`/`rebranding`/`operator_change` |

Reusing one row shape across both creates fewer required columns and more nullable ones — every analyst query becomes "filter where record_kind = X". Separate workbooks keep every column meaningful in its workbook.

## 4. The 5-sheet layout

Each workbook contains:

### 4.1 · `DATA` sheet (DATA tab named `TRANSACTIONS` or `PROJECTS`)

The append-only canonical corpus. Frozen header row, bold, dark-forest fill. Headers:

```
<domain columns...> | <ingestion-meta columns>
                     canonical_id, ingestion_id, source_file, source_kind,
                     source_url, ingested_at, ingested_by, normalization_version,
                     dedup_key, review_required, review_reason, ingestion_status,
                     supersedes_id, notes
```

Domain columns are per-workbook (transaction-schema.md / project-schema.md). The 14-column ingestion-meta block is identical across both.

### 4.2 · `DICTIONARY` sheet

One row per column. Five columns: `section`, `column`, `type`, `required`, `notes`. The contract operators consult when they don't remember whether `country` is `ES` or `Spain` (the answer: `ES`).

### 4.3 · `INGESTION_LOG` sheet

One row per processed file. Fifteen columns:

```
ingestion_id, source_file, source_kind, started_at, completed_at,
rows_seen, rows_inserted, rows_updated, rows_skipped, rows_flagged_review,
rows_failed, operator_email, normalization_version, outcome, notes
```

This is the per-workbook audit trail. The operator can reconstruct any past run's outcome without leaving Excel.

### 4.4 · `SOURCES_REGISTRY` sheet

Maps the `source_kind` enum values to labels + reliability tiers:

| source_kind | Label | Tier | Notes |
|---|---|---|---|
| costar | CoStar Export | A | Authoritative |
| brokerage | Brokerage advisor list | B | CBRE/JLL/Cushman/Christie/Colliers |
| curated | Curated internal compilation | A | Hand-maintained ground-truth |
| press_release | Official press release | B | Verify cross-reference |
| news_extract | News extraction (Intelligence Engine) | C | Validate before merging |
| manual | Manual operator entry | C | Audit via `ingested_by` |

### 4.5 · `README` sheet

In-workbook operating contract — what an operator opening the file should understand in 30 seconds.

## 5. Ingestion-meta block — the audit contract

Every row in DATA carries the same 14 columns of metadata. These are the institutional disciplines that turn a spreadsheet into a dataset:

| Column | Role |
|---|---|
| `canonical_id` | Primary key — stable across edits |
| `ingestion_id` | FK to INGESTION_LOG — links row to processing run |
| `source_file` | Verbatim original filename — provenance |
| `source_kind` | Enum from SOURCES_REGISTRY |
| `source_url` | Public URL when applicable — provenance |
| `ingested_at` | ISO-8601 UTC — when the row landed |
| `ingested_by` | Operator email — who initiated |
| `normalization_version` | Schema version applied — reproducibility |
| `dedup_key` | sha256 of normalised dedup fields — see normalisation rules |
| `review_required` | Bool — true if anything was flagged |
| `review_reason` | Short label — why flagged |
| `ingestion_status` | `ingested` / `under_review` / `superseded` / `rejected` |
| `supersedes_id` | When this row replaces an earlier canonical row |
| `notes` | Free-form operator notes |

The presence of these columns is what separates a CSV dump from an institutional dataset.

## 6. Append-only discipline

The agent NEVER overwrites a canonical row. Corrections are handled as supersedence:

```
Old row:   canonical_id=A, price_eur=120000000, ingestion_status=ingested
New row:   canonical_id=B, price_eur=125000000, ingestion_status=ingested,
           supersedes_id=A   ←── points at the old row

Auto-step: old row's ingestion_status flipped to 'superseded' (in-place update — the only
           non-append operation allowed; recorded in the run's INGESTION_LOG entry)
```

Reads SHOULD filter by `ingestion_status = 'ingested'` to get the live corpus; the full history is queryable when needed.

## 7. Dedup-key construction

Per `data-normalization-rules.md`:

- **Transactions**: `sha256(lower(asset_name) + '|' + lower(city) + '|' + (closed_at OR announced_at) + '|' + round(price_eur))`
- **Projects**: `sha256(lower(project_name) + '|' + lower(city) + '|' + announced_at + '|' + lower(developer_name))`

Two rows with the same `dedup_key` are duplicates. Same `dedup_key` + same `content_hash` (sha256 of domain columns) → silent skip. Same `dedup_key` + different `content_hash` → routed to `staging/review/`.

## 8. Phase 5 migration — XLSX → Supabase

When the corpus stabilises (Phase 5 — measured by ≥1000 rows, ≤2% manual-review rate, ≥6 months of stable schema):

1. Add `ingestion_meta` columns to `public.hotel_transactions` + `public.hotel_projects`
2. Backfill rows from the XLSX masters via a one-shot importer
3. Switch the read path: agents + Library + Underwriting read from Postgres, not from XLSX
4. The XLSX masters remain as the operator-editable canonical, with a daily reconcile cron that writes any operator-side edits into Postgres
5. Eventually: the XLSX becomes a *read-only export* — the operator edits through a UI that writes to Postgres directly

The two-step is intentional: Postgres adoption only happens after the data discipline is proven.

## 9. Schema evolution

Bump `NORMALIZATION_VERSION` in `scripts/build_masters.py` when:

- A new column is added (minor bump, e.g. `v1.0` → `v1.1`)
- An existing column changes type or required-ness (major bump, e.g. `v1.0` → `v2.0`)

Major bumps require operator awareness — the agent emits a `system_alert` event when it detects a `normalization_version` mismatch between the masters and the operator's import. The migration playbook for major bumps is:

1. Re-run `build_masters.py` — masters now declare the new schema in DICTIONARY
2. Existing rows keep their old `normalization_version` value — historical record
3. New rows get the new version — going forward
4. A migration script (Phase 5+) reconciles old rows into the new schema where mechanical

The architecture explicitly accepts that columns added late will have `null` for old rows. No backfill is required for the corpus to be valid.

## 10. Why this is institutional-grade

The pattern delivers:

- **Auditability** — every row tells who wrote it, when, from what source, with which rules
- **Reproducibility** — `python build_masters.py` regenerates the schema from a checked-in script
- **Operator agency** — no SQL required to inspect or annotate
- **Reversibility** — append-only + supersedes_id means no edit is destructive
- **Migration safety** — XLSX → Supabase is a 4-step plan, not a "throw the spreadsheet at the ORM"
- **Cost discipline** — file I/O is free; the data warehouse arrives when usage justifies it

The XLSX masters are not a temporary hack. They are the institutional contract for the next 12+ months, with a deliberate path to Postgres when the corpus has earned it.
