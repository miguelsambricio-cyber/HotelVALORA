# CoStar Master Dataset Architecture

Why four canonical .xlsx workbooks today, and how they evolve into Supabase tables tomorrow.

**Last refreshed:** 2026-05-14

> **2026-05-14 — Two datasets, not one.** The COSTAR workspace now models **two genuinely distinct datasets** both sourced from CoStar exports:
>
> - **Dataset A · Market Performance** — aggregated KPIs by period (occupancy · ADR · RevPAR · room nights · supply · demand · pipeline). Granularities: country (`PAIS`), market (`MERCADO`), submarket (`SUBMERCADO`).
> - **Dataset B · Hotel-by-Market Inventory** — individual property records with slowly-changing attributes (name · brand · operator · facilities · score · category · rooms · geo). Granularity: hotel-by-hotel within a market (`HOTELES POR MERCADO`).
>
> The legacy CLASS master (chain-scale aggregates) is retired — `chain_scale` is now an attribute on each hotel record in Dataset B. See `costar-hotels-by-market-schema.md`.

---

## 1. The decision

HOTELVALORA needs an institutional, version-controlled, human-editable corpus for CoStar / STR hospitality market intelligence **before** the runtime read path is wired into Supabase. We pick:

- **Four separate .xlsx workbooks** as the canonical source — country, market, submarket, class
- **Generated by a checked-in Python script** (`services/costar/scripts/build_masters.py`) so the schema is reproducible
- **Append-only** — never overwrite; supersede via a new row with `supersedes_id`
- **Five sheets per workbook** — DATA · DICTIONARY · INGESTION_LOG · SOURCES_REGISTRY · README
- **Backed by Supabase tables in Phase 5** once enrichment + entity resolution stabilise

**Note (v1.1):** CompSet datasets moved to `services/compset/` (operational workspace, not warehouse). See `docs/architecture/market-vs-underwriting-separation.md`.

Same institutional pattern as the transactions workspace (`services/transactions/`): pen the contract in something operators can audit (Excel), enforce shape via a generator (Python), and migrate the read path to a database (Postgres) once the corpus has earned it.

## 2. Why .xlsx, not Supabase, today

Same rationale as the transactions workspace:

| Reason | Detail |
|---|---|
| **Operator legibility** | Analysts read CoStar / STR in Excel natively. Forcing SQL on the data-quality loop slows institutional adoption. |
| **Cell-level audit** | A row with a wrong RevPAR → operator opens Excel, edits, saves. Familiar workflow. |
| **No premature schema lock-in** | The first 90 days of multi-source ingestion (CoStar + STR + Kalibri + curated) will reveal columns we haven't anticipated. Excel tolerates this; Postgres makes evolution 3× more expensive. |
| **Reproducible** | The generator is the contract. Anyone can rebuild the workbook from `build_masters.py`. |
| **Git-friendly for the canonical** | Four .xlsx blobs in git are ~50 KB combined. |
| **Zero infrastructure** | No migrations, no RLS, no service-role keys, no API surface. Phase 1 ships with file I/O only. |

## 3. Why four workbooks, not one

The four granularities diverge on multiple axes — collapsing them forces nullable columns everywhere and ambiguous primary keys. Each workbook stands on its own:

| Dimension | Country (PAIS) | Market (MERCADOS) | Submarket (SUBMERCADOS) | Hotels by Market (HOTELES POR MERCADO) |
|---|---|---|---|---|
| **Dataset** | A · market performance | A · market performance | A · market performance | B · hotel inventory |
| **Primary key** | (country, period, period_kind) | (country, market_name, period, period_kind) | (country, market_name, submarket_name, period, period_kind) | (country, market_name, hotel_id) |
| **Time series** | yes (quarterly / annual) | yes (monthly / quarterly) | yes (monthly) | no — slowly-changing dimension |
| **Macro context columns** | yes (GDP, inflation, tourism arrivals) | no | no | n/a |
| **Positioning indicators** | n/a | revpar_index_vs_country | revpar_index_vs_market | derived via JOIN to market metrics |
| **Aggregation rules** | source-direct | hierarchical (markets sum into country) | hierarchical (submarkets sum into market) | n/a — atomic records |
| **Underwriting role** | macro substrate | asset-anchor | comp neighborhood | **reference data backbone** (compsets · valuations · benchmarks) |
| **~~Class (legacy)~~** | retired 2026-05-14 — `chain_scale` is now an attribute on each hotel record in Dataset B |

Reusing one row shape across all four creates fewer required columns and more nullable ones — every analyst query becomes "filter where granularity = X". Separate workbooks keep every column meaningful in its workbook.

**CompSet datasets are NOT in this warehouse.** They live in `services/compset/` because they are operational, hotel-specific, and underwriting-critical — fundamentally different from the warehouse rhythm. See `docs/architecture/market-vs-underwriting-separation.md`.

## 4. The 5-sheet layout (identical across all 4 workbooks)

### 4.1 · `DATA` sheet (tab named `COUNTRY` / `MARKET` / `SUBMARKET` / `COMPSET`)

Append-only canonical corpus. Frozen header row, bold, dark-forest fill. Headers:

```
<domain columns...> | <ingestion-meta columns>
                     canonical_id, ingestion_id, source_file, source_kind,
                     source_url, ingested_at, ingested_by, normalization_version,
                     dedup_key, review_required, review_reason, ingestion_status,
                     supersedes_id, notes
```

The 14-column ingestion-meta block is **identical** across all four workbooks AND across the `transactions/` masters — a deliberate consistency that lets the Data Ingestion Agent reuse the same routing logic in both workspaces.

### 4.2 · `DICTIONARY` sheet

One row per column. Five columns: `section`, `column`, `type`, `required`, `notes`. The contract operators consult when they don't remember whether `country` is `ES` or `Spain` (the answer: `ES`).

### 4.3 · `INGESTION_LOG` sheet

One row per processed file. Fifteen columns: `ingestion_id`, `source_file`, `source_kind`, `started_at`, `completed_at`, `rows_seen`, `rows_inserted`, `rows_updated`, `rows_skipped`, `rows_flagged_review`, `rows_failed`, `operator_email`, `normalization_version`, `outcome`, `notes`.

This is the per-workbook audit trail. Operator can reconstruct any past run's outcome without leaving Excel.

### 4.4 · `SOURCES_REGISTRY` sheet

Maps the `source_kind` enum values to labels + reliability tiers:

| source_kind | Label | Tier | Notes |
|---|---|---|---|
| costar | CoStar Hospitality Export | A | Authoritative — institutional, paid product. The canonical source. |
| str | STR (CoStar subsidiary) | A | STR-direct exports — same provenance as CoStar. |
| kalibri | Kalibri Labs export | B | Operator analytics — useful for cross-validation. |
| curated | Curated internal compilation | A | Hand-maintained ground-truth. |
| manual | Manual operator entry | C | Operator-typed row. |

### 4.5 · `README` sheet

In-workbook operating contract — what an operator opening the file should understand in 30 seconds.

## 5. Ingestion-meta block — the audit contract

Every row in every DATA sheet carries the same 14 metadata columns. Reused identically from the `transactions/` masters by design — the Data Ingestion Agent treats both workspaces with the same routing + dedup + audit shape.

| Column | Role |
|---|---|
| `canonical_id` | Primary key — stable across edits |
| `ingestion_id` | FK to INGESTION_LOG — links row to processing run |
| `source_file` | Verbatim original filename — provenance |
| `source_kind` | Enum from SOURCES_REGISTRY |
| `source_url` | Public URL when applicable |
| `ingested_at` | ISO-8601 UTC — when the row landed |
| `ingested_by` | Operator email — who initiated |
| `normalization_version` | Schema version applied — reproducibility |
| `dedup_key` | sha256 of normalised dedup fields |
| `review_required` | Bool — true if anything was flagged |
| `review_reason` | Short label — why flagged |
| `ingestion_status` | `ingested` / `under_review` / `superseded` / `rejected` |
| `supersedes_id` | When this row replaces an earlier canonical row |
| `notes` | Free-form operator notes |

## 6. Append-only discipline

The agent NEVER overwrites a canonical row. Corrections are handled as supersedence:

```
Old row:   canonical_id=A, revpar=121.50, ingestion_status=ingested
New row:   canonical_id=B, revpar=124.20, ingestion_status=ingested,
           supersedes_id=A   ←── points at the old row

Auto-step: old row's ingestion_status flipped to 'superseded' (in-place update — the only
           non-append operation allowed; recorded in the run's INGESTION_LOG entry)
```

Reads SHOULD filter by `ingestion_status = 'ingested'` to get the live corpus; the full history is queryable when needed.

For CoStar specifically, **monthly revisions are normal** — STR/CoStar regularly restate prior periods as late data arrives. Each restatement becomes a new canonical row with `supersedes_id` pointing at the prior version. The full restatement history is preserved.

## 7. Dedup-key construction

Per `costar-normalization-rules.md` §4:

- **Country**: `sha256(country | iso_date(period_start) | period_kind)`
- **Market**: `sha256(country | normalised(market_name) | iso_date(period_start) | period_kind)`
- **Submarket**: `sha256(country | normalised(market_name) | normalised(submarket_name) | normalised(chain_scale) | iso_date(period_start) | period_kind)`
- **Compset**: `sha256(normalised(compset_name) | normalised(target_hotel_name) | iso_date(period_start) | period_kind)`

Two rows with the same `dedup_key` are the same period × entity. Same `dedup_key` + same `content_hash` → silent skip. Same `dedup_key` + different `content_hash` → routed to `staging/review/` for the operator to resolve (typically a CoStar restatement).

## 8. Source-tier precedence (when reviewing duplicates)

| Tier | Sources | Precedence |
|---|---|---|
| A | `costar`, `str`, `curated` | Wins over B and C |
| B | `kalibri` | Wins over C |
| C | `manual` | Lowest |

Within the same tier, the more recent `ingested_at` wins. Tie-break: the row with fewer null domain columns wins. The reviewer can always override.

## 9. Phase 5 migration — XLSX → Supabase

When the corpus stabilises (Phase 5 — measured by ≥36 months of country data, ≥12 months of compset data per active hotel, ≤2% manual-review rate, ≥6 months of stable schema):

1. Create `public.market_periods` table covering country + market + submarket time series with a `granularity` enum and a polymorphic FK to `public.markets` + `public.submarkets` reference tables
2. Create `public.compset_periods` table for the compset-level data (different shape — subject + compset KPIs + indices)
3. Backfill rows from the XLSX masters via a one-shot importer
4. Switch the read path: Underwriting Engine + Library + Market Overview reports read from Postgres, not from XLSX
5. The XLSX masters remain operator-editable canonical with a daily reconcile cron
6. Eventually the XLSX becomes a read-only export

The two-step is intentional: Postgres adoption only happens after the data discipline is proven.

## 10. Cross-workspace consistency with `services/transactions/`

The two workspaces share architectural primitives by design:

| Primitive | Same? |
|---|---|
| 14-column ingestion-meta block | ✅ identical |
| 5-sheet workbook layout | ✅ identical |
| `.gitignore` posture (track contract, not data) | ✅ identical |
| `scripts/build_masters.py` pattern | ✅ identical |
| Data Ingestion Agent routing logic | ✅ shared (Phase 2.3.d wires costar with the same primitives as 2.3.b wired transactions) |
| Audit-chain unification via `/api/agents/data-ingestion-summary` | ✅ shared cloud endpoint |
| SOURCES_REGISTRY | ⚪ different vocab (transactions: costar/brokerage/curated/news_extract/manual/press_release; costar: costar/str/kalibri/curated/manual) |

This consistency means a Phase 5 Postgres migration migrates **both** workspaces at once with shared infrastructure.

## 11. Schema evolution

Bump `NORMALIZATION_VERSION` in `scripts/build_masters.py` when:

- **Minor bump (`v1.0 → v1.1`)**: a new column added (e.g. occupancy by chain scale at country level)
- **Major bump (`v1.0 → v2.0`)**: existing column changes type, required flag, or unit (e.g. switching ADR from period-average to room-night-weighted)

Major bumps require operator awareness — the agent emits a `system_alert` event when it detects a `normalization_version` mismatch between the masters and the operator's import.

## 12. Why this is institutional-grade

- **Auditability** — every row tells who wrote it, when, from what source, with which rules version
- **Reproducibility** — `python build_masters.py` regenerates the schema deterministically
- **Operator agency** — underwriters edit + inspect in Excel
- **Reversibility** — append-only + supersedes_id means no edit is destructive
- **Migration safety** — XLSX → Supabase is a multi-step plan with parallel running
- **Cost discipline** — file I/O is free; the data warehouse arrives when usage justifies it
- **Cross-workspace consistency** — `transactions/` + `costar/` share the same primitives so one Agent + one cloud endpoint supervise both

The XLSX masters are not a temporary hack. They are the institutional contract for the next 12+ months, with a deliberate path to Postgres when the corpus has earned it.
