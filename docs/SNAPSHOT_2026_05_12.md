### 3.4 · T1.5 → T2 session architecture (Option B)

**Decision history:** Originally approved as Option A (T1 in Vercel env vars only). Pivoted to Option B (T1 encrypted in Supabase) on 2026-05-12 because operator console UX trumped the slight risk of T1 ciphertext in Postgres (KEK still env-only · same threat model as T2).

**Live state:**

```
T1 · Raw credentials
   storage:  public.intelligence_source_credentials
             AES-256-GCM with separate IV per field (username + password)
             KEK: INTELLIGENCE_SESSION_ENC_KEY (32B base64, env)
   write:    /user/admin/integrations/[id] → CredentialsPanel form
             → provisionCredentialsAction (server action · admin gate)
             → server-side encrypt with redact()
             → upsert + audit_event 'provisioned'
   read:     getCredentialsStatus(slug) returns NON-SECRET status only
             (configured/status/last_login_at/...)
             plaintext only re-emerges in getDecryptedCredentials(slug)
             which is called by the refresh script context
   audit:    public.intelligence_credentials_audit · append-only
             events: provisioned · rotated · invalidated · auth_success
                     · auth_failure · decryption_error

T2 · Session artifact
   storage:  public.intelligence_source_sessions
             AES-256-GCM(storageState JSON) · independent IV
             enc_key_id tracks the wrapping KEK (rotation support)
   write:    apps/web/scripts/execute-session-refresh.mjs (one-off CLI)
             → reads T1 ciphertext, decrypts, verifies round-trip
             → builds PLACEHOLDER storageState (real Playwright capture
               is Phase 2.5b)
             → encrypts and persists · 7-day TTL · audit event
             → updates credentials.last_login_at = success
   read:     reserved for the ingestion run context (Phase 2.5b will
             pass the storageState to Playwright/cookie-jar fetch)
   placeholder marker: storageState.meta.placeholder = true
                       cookies named "__placeholder_<slug>_session"
```

**RLS posture (defence-in-depth):**
- All three tables: RLS enabled · zero policies (no anon / authenticated SELECT)
- `revoke all on … from anon, authenticated` (belt-and-braces)
- Only service-role bypasses RLS → only trusted server contexts can read

### 4.3 · Placeholder session architecture (Phase 2.5)

**Purpose:** Stand up the entire T1 → T2 → ingestion → audit lifecycle without yet implementing Playwright. Validates the security architecture end-to-end (encryption round-trip · audit chain · dashboard inference) before committing to the heavier Playwright wiring.

**Construction:**

```js
// apps/web/scripts/execute-session-refresh.mjs
// 1. Decrypt T1 credentials (proves AES-256-GCM round-trip)
// 2. Build placeholder storageState:
const storageState = {
  cookies: [{
    name: `__placeholder_${slug}_session`,
    value: `pending-playwright-script-${nowIso}`,
    domain: `.${sourceDomain}`,
    ...
  }],
  origins: [],
  meta: { placeholder: true, captured_at, note: '...' }
};
// 3. AES-256-GCM(JSON.stringify(storageState))
// 4. UPSERT intelligence_source_sessions · status=active · 7d TTL
// 5. UPDATE intelligence_source_credentials.last_login_status='success'
// 6. INSERT intelligence_credentials_audit · event_kind='auth_success'
//    with detail.placeholder_storage_state = true
```

**How to distinguish a placeholder from a real session:** `meta.placeholder === true` OR cookie name matches `__placeholder_*`. The Phase 2.5b refresh script will overwrite these rows with real captures (same wire format · independent IV).

### 4.4 · Planned Playwright implementation (Phase 2.5b)

**Goal:** Replace placeholder sessions with real authenticated logins against `hosteltur.com` and `alimarket.es` using Playwright + `playwright-core` + `@sparticuz/chromium` (Vercel-friendly bundle).

**Architecture:**

```
operator triggers refresh from /user/admin/integrations/[id]
   → server action /api/agents/refresh-session?slug=...
   → calls Playwright (or playwright-core + sparticuz/chromium)
   → load login page · fill creds · submit · wait for post-login marker
   → context.storageState() captures cookies + localStorage
   → AES-256-GCM(storageState) + UPSERT intelligence_source_sessions
   → audit event auth_success (placeholder flag absent)
```

**Open questions for Phase 2.5b:**
- Run Playwright in Vercel function (size constraints · ~250MB chromium binary) or in a separate worker (Render / Fly)?
- Reuse the existing `execute-session-refresh.mjs` script with a `--mode=playwright` flag, or split into two scripts?
- Handle MFA / CAPTCHA fallback paths?
- Rate-limiting between refresh attempts (Hosteltur subscription rate limits?)

**Body-fetch pipeline (parallel work):**
- For each `market_news.url` with `raw_meta.fetch_kind='public_preview'` or `'public_rss'`, the ingestion run uses the decrypted session cookies to fetch the full body, parse it, and update `market_news.body` / `body_text` / `enriched_meta`.

### 4.6 · CompSet operational strategy

**Decision:** CompSet construction is operator-curated per valuation. The agent (`compset_underwriting`) does not auto-build comparator sets — it surfaces a recommended set from:
- `services/costar/MASTER/SUBMERCADO` (the market context)
- `public.valuations` (the subject hotel)
- Operator's investment criteria (`/settings/investment/*`)

The agent's output is a draft CompSet that the operator reviews and confirms in the Underwriting workbench. The `services/compset/MASTER/` Excel masters track the final operator-confirmed sets per valuation. See `docs/architecture/market-vs-underwriting-separation.md`.
